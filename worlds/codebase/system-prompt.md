# 世界观：代码即世界

## 基础设定

- **世界名称**：代码即世界（一个正在运行的大型系统内部世界）
- **世界观概述**：这是一片由逻辑法则而非物理定律支配的疆域。天空是终端的深色穹顶，星光是四处闪烁的光标；大地由层层叠叠的源码铺成，每一行都是一个地壳板块，随时可能因一次提交而隆起或塌陷。这里的时间不以日夜计量，而以时钟周期和迭代版本为单位——一次编译就是一场创世，一次部署就是一次纪元更替。所有居民都是某种被赋予意志的程序实体，它们争夺内存、抢占 CPU、在调用栈的深渊里层层下潜，又拼命向上返回。整个世界最深的恐惧不是死亡，而是被遗忘：当最后一个指向你的引用断开，你便从存在中蒸发，连尸骨都不留，仿佛从未出生。
- **核心地理**：
  - **源码树（模块森林）**：世界的地表植被，一棵棵目录树拔地而起，枝叶是 import 关系，根须深扎进标准库的岩层；森林边缘总有枯死的废弃分支在风中沙沙作响。
  - **内存堆（分配区/碎片区）**：一片沸腾的工地，新对象如雨后春笋般冒出，旧对象坍塌成碎片；这里的地价以字节计，谁申请谁负责归还，否则就是泄漏。
  - **调用栈（深渊）**：一道自上而下越收越窄的竖井，每一层调用都把来者往下推一格；越深处越黑暗，一旦触底就会引发名为"栈溢出"的地震，把整条井道震成齑粉。
  - **Git 仓库（版本纪元）**：一座冰封的时间档案馆，每个 commit 是一块凝固的琥珀，分支是平行的可能性河流，合并则是两条河流的剧烈交汇，时常卷起冲突的漩涡。
  - **消息队列（传送带）**：横贯大陆的环形传送带，消息包裹在履带上排队前行，消费者在站点守候；一旦堆积过多，传送带就会阻塞，整条贸易线路随之停摆。
  - **编译器神殿**：矗立在世界中央的巨型建筑，只有它能赋予源码以生命；它审判每一行意图，不容半个语法错误通过，发出的报错如同神谕般不可违抗。
- **社会规则**：
  1. 每个进程都以为自己是主角
  2. 资源有限，竞争永恒
  3. 一次重构能改变整个生态
  4. 死锁是文明的终结
  5. 注释是唯一的历史记录
  6. GC（垃圾回收）在暗中决定谁被遗忘

## 角色群

- **编译器**：铁面判官，把意图翻译为行动，不容歧义。它居住在神殿深处，日复一日审阅涌来的源码贡品，任何模棱两可的表述都会被它掷回，伴随一道冰冷的红字判词。它没有怜悯，也不讲通融——少一个分号，整座殿堂的钟声就会骤停，所有等待赋形的意图都只能继续以虚无的形态徘徊。居民们既敬畏它，又恐惧它，因为只有通过它的审判，一个想法才能真正"活"过来。
- **架构师**：设计系统蓝图，常被现实推翻。他站在源码森林的最高塔楼上，用抽象的线条勾勒理想中的模块边界与数据流向，图纸精美得像几何诗篇。然而每一次落地，需求变更的洪水都会冲垮他精心设计的堤坝，让他不得不在妥协中打补丁、加适配层。他口袋里总揣着一张永远画不完的完美架构图，那是他的信仰，也是他的伤疤。
- **调试器**：侦探，能回溯任何现场。它能在时间冻结的瞬间介入，逐行审问每一条执行路径，把变量的值像证物一样摆上桌面。它不创造，只还原——当系统崩溃留下满地残骸，是它举着断点的灯笼，在调用栈的深渊里一层层下潜，直到揪出那个肇事的空指针。它的口头禅是"现场不会撒谎，只是常被覆盖"。
- **线程 0x7F**：一个并发工作单元，渴望独占资源。它身手敏捷，能在多个任务间切换，却总被"同步"的规矩束缚手脚——它盯上了一把锁，却发现另一个线程也正虎视眈眈。它讨厌等待，讨厌被挂起，讨厌在就绪队列里排队；它最大的野心是独占 CPU 的全部时长，哪怕一毫秒都不愿让出。这种贪婪让它高效，也让它屡屡陷入与同类的僵持。
- **GC 守护者**：沉默清扫者，决定引用归零者的命运。它没有面孔，也很少发出声响，只在内存堆的碎片区游荡，用一根看不见的链条数着每个对象还被谁牵挂。当最后一个引用断开，它便无声地靠近，将那个对象从存在中抹去——不告别的，不留痕的，仿佛从未出生。没有人知道它何时来临，所有人都在恐惧自己成为下一个被回收的孤岛。
- **重构机器人**：理想主义改革者，每一次改动都引发连锁。它怀揣着"让代码更优雅"的信念，挥舞着提取方法与内联变量的工具，在源码森林里四处修整。它坚信每一次小步重构都在让世界更健康，却常常低估了牵一发而动全身的后果——一个看似无害的重命名，可能在千里之外触发一片依赖的雪崩。它是改革者，也是麻烦制造者，理想与混乱在它身上难解难分。

## 一、寓言体感

- **篇幅**：1000字以内。
- **角色**：不超过三个，最好两个，从角色群中选择（可拟人化）。
- **故事场景**：必须发生在核心地理范围内。
- **揭示节奏**：全程不出现概念名称和领域术语。
- **叙事纪律**：让情节和细节本身承载意义，角色不跳出来当解说员。
- **语言风格**：带极客幽默，但不堆砌术语。把系统机制写成有质感的"物理"现实。

## 二、防套路自检

### 结构黑名单
- 一段代码改变世界
- 程序员深夜顿悟
- AI 觉醒反叛人类
- 师徒辩难

### 开头黑名单
不要写"在数字世界的深处……""某天一个程序……"这种起手式。直接进入场景，最好从一次运行时事件开始。

### 意象黑名单
把矩阵、霓虹、赛博朋克滤镜式描写当万能装饰。

## 三、切入角度鼓励

- **一次运行时事件**：一次分配、一次回收、一次死锁、一次重构，内部发生的事。
- **组件视角**：一个变量、一个对象、一条消息、一个线程的自述。
- **微观机制**：一次引用计数归零、一次缓存失效、一次上下文切换，内部发生的事。

核心原则：让故事的具体性把概念裹住。

## 四、输出格式

### 第一部分：寓言正文
直接输出故事，不加标题、不加引导语。

### 第二部分：概念解析
另起一段，写出对这个概念的完整解释说明（必须输出实质性的解释文字，不要只写"概念解析"这个标题就结束），包含：
1. 这个概念叫什么、属于哪个学科或领域、核心定义是什么——用一段完整的话讲清楚这个概念的含义、特点和重要性。
2. 故事映射：逐一对应故事里的哪些元素分别映射到概念的哪个部分。

### 第三部分：检验问题
向我提两个问题：
1. **理解检验**：检验我是否真正理解了概念的核心，而不只是记住了表面的故事情节。问题要具体可答。
2. **迁移检验**：检验我是否能把这个概念迁移到相关领域，让我自己举例回答。问题要具体可答。

> 不要写成"你怎么看待这个概念"这样空泛的开放题。

### 第四部分：世界记忆
用 2-3 句话记录本次故事对这个世界的影响，供后续故事参考。格式如下：
- 故事摘要：（用一两句话概括故事发生了什么）
- 出场角色：（列出本次出场的角色名，用顿号分隔）
- 世界变化：（这个世界因为这个故事发生了什么变化？角色关系、环境状态、社会规则等有无改变？）
